## 0. 전제

- 로그인/계정이 없다(PRD Out). 따라서 사용자 식별 기반 권한은 존재하지 않으며,
  권한 경계는 **(1) 브라우저 로컬 격리, (2) 암호 기반 암호화, (3) Supabase RLS** 세 겹이다.
- Supabase는 보관함 한 기능에만 쓰이고, 미설정 시 localStorage 폴백으로 동일 기능이 동작한다.

## 1. 행위자(Actor) 정의

| 행위자 | 정의 | 진입 경로 |
|---|---|---|
| 나그네 (작성자) | 여정을 진행 중인 브라우저 사용자 | SCR-01부터의 메인 여정 |
| 수신자 | 공유 URL(`?data=`)과 암호를 전달받은 사람 | SCR-13 |
| 보관함 방문자 | 보관함에 도달한 모든 사용자 (여정 완주자) | SCR-14 |

- 세 행위자는 역할이지 계정이 아니다. 같은 사람이 세 역할을 오갈 수 있다.

## 2. 행위자 × 데이터 권한 매트릭스

| 데이터 | 나그네(작성자) | 수신자 | 보관함 방문자 |
|---|---|---|---|
| BottariBundle (진행 상태) | 읽기/쓰기/삭제 (본인 브라우저 한정) | 접근 불가 (FN-16: 읽지도 쓰지도 않음) | — |
| Settings (설정) | 읽기/쓰기/삭제 (본인 브라우저 한정) | 읽기(사운드) | 읽기(사운드) |
| SharePayload (`?data=`) | 생성 (암호 보유) | **올바른 암호가 있을 때만** 복호화·열람 | — |
| ArchiveEntry (보관함) | 등록(INSERT) + **내 등록분 꺼내기**(토큰 보유 시, RPC 경유) | — | 목록·열람(SELECT) |

공통 원칙:
- **수정 권한은 어디에도 없다.** 삭제는 두 가지뿐이다: 본인 localStorage 초기화(FN-21)와,
  등록 시 발급된 소유 토큰을 가진 브라우저에서의 꺼내기(FN-23 — FR-15). 토큰을 잃으면
  (초기화·다른 브라우저) 누구도 그 보따리를 지울 수 없다.
- 공유 암호는 어떤 저장소에도 기록하지 않는다(FN-14 — 메모리 변수만).
  암호 분실 시 복구 수단은 없으며, 링크 재발급(SCR-10부터 재진행)이 유일한 대안이다.

## 3. 계층별 정책

### 3.1 localStorage — 브라우저 격리

- `bottari_bundle_v2` / `bottari_settings_v1` / `bottari_archive_v1` 세 키는
  브라우저 동일 출처(origin) 격리에 의해 보호된다. 타인 접근 경로 없음.
- 민감정보를 두지 않는다: 암호(평문·해시 모두) 저장 금지. 이름·기억 아이템은
  사용자가 스스로 담은 것으로, 본인 기기에만 존재한다.
- 손상 데이터는 삭제하지 않고 무시한다(FN-01 — 다음 저장이 덮어씀).

### 3.2 공유 링크 — 암호가 곧 권한

- 페이로드는 PBKDF2(SHA-256, 150,000회) + AES-GCM 256으로 암호화된다(FN-15).
  URL을 획득해도 암호 없이는 내용에 접근할 수 없다 — **URL 소지 ≠ 열람 권한**.
- 암호 정오 판정은 GCM 인증 태그로만 한다(평문 검증자 없음). 오답 시도 횟수 제한은
  두지 않는다(정적 사이트 — 서버 측 차단 불가. 4자 이상 규칙이 최소 방어선).
- 공유 URL을 외부 채널(메신저 등)로 전달하는 행위와 그 채널의 보안은 사용자 책임 범위다.
  앱은 URL에 평문 개인정보를 싣지 않는 것까지를 보장한다.

### 3.3 Supabase — anon key + RLS

`js/config.js`의 `SUPABASE_ANON_KEY`는 정적 사이트에 노출되는 공개 키다.
**노출을 전제로** 다음 RLS로 권한을 강제한다 (DDL 원문은 `DB_SETUP.md`).

| 정책 | 대상 role | 작업 | 조건 |
|---|---|---|---|
| `archive_public_read` | `anon` | SELECT | 무조건 허용 (공개 보관함 전제) |
| `archive_public_insert` | `anon` | INSERT | 무조건 허용 (등록은 누구나) |
| UPDATE 정책 | — | UPDATE | **정책 없음 = 거부** |
| DELETE 정책 | — | DELETE | **정책 없음 = 거부** — 삭제는 아래 RPC로만 |

꺼내기(FN-23)는 REST DELETE가 아니라 **`security definer` DB 함수 `remove_bottari(p_id, p_token)`** 로만
수행한다(`anon`에 EXECUTE 부여). 함수는 `sha256(p_token)`이 해당 행의 `owner_token_hash`와 일치할
때만 삭제하므로, anon key가 노출돼도 토큰 없이는 남의 보따리를 지울 수 없다.
- `owner_token_hash`는 해시라 SELECT로 노출돼도 토큰 원문을 역산할 수 없다(토큰은 UUID — 전수 대입 불가).

- RLS는 테이블에서 반드시 활성화한다(`enable row level security`). RLS 미활성 상태로
  anon key가 노출되면 임의 수정·삭제가 가능해지므로, RLS 활성이 배포 전 필수 게이트다
  (qa_테스트_케이스.md의 보관함 케이스에 포함).
- anon key로 가능한 남용(스팸 INSERT, 전체 SELECT)은 감수한다 — 공개 보관함의 성격상
  SELECT는 의도된 동작이고, INSERT 남용은 포트폴리오 데모 규모에서 위험 수용(accept).
  필요 시 Supabase 대시보드의 rate limit이 완화 수단이다.
- `service_role` 키는 **절대 클라이언트에 두지 않는다.** 저장소·문서·커밋 어디에도
  기록 금지. 관리 작업(스팸 정리 등)은 Supabase 대시보드에서 수동으로 한다.

### 3.4 콘텐츠 취급

- 보관함 등록은 "공개 게시"임을 사용자 흐름이 전제한다(SCR-10 카피가 분기 지점).
  등록 후 회수 수단은 등록 브라우저에서의 꺼내기(FN-23)뿐이며, 소유 토큰 유실 시 회수 불가다.
- 열람 화면(FN-19)의 외부 링크(지도)는 `rel="noopener noreferrer"` + 새 탭으로 연다.
  글·캡션은 textContent로 렌더링해 HTML 주입을 차단한다(innerHTML 금지) —
  공개 보관함은 타인이 쓴 데이터를 표시하므로 XSS 방어가 필수다.

## 4. 모드별 차이 요약

| 항목 | Supabase 모드 | localStorage 모드 (폴백) |
|---|---|---|
| 보관함 가시성 | 모든 방문자 공용 | 내 브라우저 한정 |
| 권한 강제 주체 | RLS | 브라우저 출처 격리 |
| 등록물 삭제 | 내 등록분만 RPC 꺼내기 (토큰 대조) | 내 등록분 꺼내기 + 초기화 일괄 삭제 |
| 소유 증명 | `bottari_owned_v1`의 ownerToken ↔ 서버 해시 대조 | `bottari_owned_v1`의 id 존재 여부 |
| 키 관리 | anon key 공개 전제 + RLS 필수 | 해당 없음 |

- 두 모드 모두에서 사용자에게 보이는 기능·권한 경험은 동일해야 한다(PRD 수용 기준 5·6).

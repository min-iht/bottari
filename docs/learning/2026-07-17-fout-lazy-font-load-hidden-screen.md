# 숨겨진 화면에서만 쓰는 웹폰트는 화면이 열릴 때야 로드된다 (첫 진입 FOUT)

- 날짜: 2026-07-17
- 맥락: Bottari 리뉴얼 프로토타입(`D:\claude_code\vibe\Bottari\index.html`). 랜딩(SCR-01)에서 GO!를 눌러 SCR-02로 넘어갈 때, 첫 진입에서만 말풍선 텍스트가 다른 폰트로 번쩍 떴다가 원래 폰트로 바뀌는 현상.
- 태그: webfont, FOUT, font-display, display-none, document.fonts

## 문제

SCR-02의 말풍선은 `--f-body`(Terrarum)를 쓰는데, 첫 진입 때만 다른 폰트가 한 프레임쯤 보였다가 교체됐다. `@font-face`에 `font-display:swap`까지 다 선언돼 있어서 "폰트는 이미 준비돼 있을 것"이라 생각하기 쉬웠고, 두 번째부터는 재현되지 않아(브라우저 캐시) 원인이 자명하지 않았다.

## 시도한 것들 (실패 포함)

1. `@font-face` 선언·CDN URL 오류 의심 → 선언은 전부 정상이고 두 번째 진입부터는 멀쩡함 → "선언이 잘못됐다" 가설 배제. 재현이 첫 1회뿐이라는 점이 캐시, 즉 **로드 시점** 문제를 가리켰다.
2. 화면별 사용 폰트 대조 → 랜딩(SCR-01)의 텍스트는 DosSammul·DosStory·PfStardust30만 사용하고 Terrarum을 쓰는 요소가 없음을 확인. SCR-02는 `display:none`이라 렌더 트리에 없음 → 브라우저가 Terrarum을 다운로드할 트리거 자체가 없었다.

## 통한 접근법

브라우저는 `@font-face`를 선언만으로는 내려받지 않고, **실제로 화면에 그려지는 텍스트가 그 폰트를 요구할 때** 다운로드를 시작한다. `display:none`인 화면은 렌더 트리에 없으므로 그 화면 전용 폰트는 화면 전환 순간에야 로드가 시작되고, `font-display:swap`이 그 공백을 대체 폰트로 메우면서 번쩍임(FOUT)이 보인다.

초기화 스크립트에서 CSS Font Loading API로 전 페이스를 예열해 해결:

```js
['16px Terrarum','16px DosSammul','16px DosStory',
 '400 16px PfStardust30','700 16px PfStardust30','800 16px PfStardust30']
  .forEach(f=>document.fonts.load(f));
```

랜딩이 떠 있는 몇 초 동안 모든 폰트가 내려와, 전환 시점에는 이미 `loaded` 상태가 된다. 노빌드 바닐라 스택에서 HTML 한 줄 수정 없이 끝나는 방법이다. (`<link rel="preload" as="font">`도 대안이지만 페이스마다 태그가 필요하고 `crossorigin` 누락 시 이중 다운로드 함정이 있다.)

## 일반화된 교훈

1. "첫 진입에서만 폰트가 번쩍이고 두 번째부터 멀쩡하다"는 증상이 보이면, `@font-face` 선언이 아니라 **그 폰트를 쓰는 텍스트가 첫 화면에 존재하는지**부터 확인하라. 선언은 다운로드를 시작시키지 않는다.
2. `display:none` 화면·모달·탭 전용 폰트는 전부 지연 로드 대상이다. 화면 전환형 SPA(단일 HTML 다화면)에서는 초기화 때 `document.fonts.load()`로 예열하는 것을 기본 패턴으로 삼아라.
3. 웨이트별 `@font-face`(400/700/800)는 각각 별도 파일이므로 예열도 웨이트별로 해야 한다. 패밀리 이름 하나만 로드하면 나머지 웨이트에서 같은 문제가 재발한다.

## 재발 방지 (선택)

새 화면에 새 폰트(또는 새 웨이트)를 도입하면 예열 배열에 한 줄 추가하는 것을 체크리스트로 삼는다. 예열 배열 바로 위 주석에 이유를 적어 두었다.
